## 付録 A: Issue #1 全文

> **Title**: ネストした value 駆動 AnimationScope で祖先スコープのアニメーションが無音で strip される (failing test 付き)
> **Labels**: bug

## 概要

value 駆動の `AnimationScope` をネストすると、**外側スコープ自身の value 駆動アニメーションが、内側スコープの境界で無音で strip される**。リーク検出器はスタンプ済みトランザクションを報告しない設計のため、この消失は DEBUG 診断にも現れない。

実アプリ (NumPath) への導入で「同一サブツリーに 2 つのトリガー (セル選択 = easeOut 0.12 / ヒント表示 = spring)」をネストで表現したところ、外側の選択アニメーションが死んでいることに静的検査・スクリーンショット検証・`detectAnimationLeaks()` のいずれも気づけず、境界実装のソースリーディングと下記の failing test で初めて確定した。

## 再現 (failing test)

上流のテストハーネス (`TransactionRecorder` / `host` / `pumpRunLoop`) だけで再現できる。

```swift
private struct NestedView: View {
  @ObservedObject var model: Model
  let recorder: TransactionRecorder

  var body: some View {
    AnimationScope(.easeOut(duration: 0.12), value: model.outerTrigger, name: "Outer") {
      AnimationScope(.spring(response: 0.35, dampingFraction: 0.7), value: model.innerTrigger, name: "Inner") {
        Text("nested")
          .opacity(model.outerTrigger ? 0.2 : 1)
          .transaction { recorder.record("nested-child", $0) }
      }
    }
  }
}

// model.outerTrigger.toggle() 後:
XCTAssertTrue(recorder.hasAnimation("nested-child"))  // ← 失敗する
```

実行結果 (macOS 26 / Swift 6 toolchain, `swift test`):

```text
CONTROL_OUTER_TRIGGER: seq=2 label=control-child animation=true  stampName=Control ...   // 非ネストの対照群は届く
NESTED_OUTER_TRIGGER:  seq=2 label=nested-child  animation=false stampName=Outer
                       stampAnimationDescription=BezierAnimation(duration: 0.12, ...)    // stamp は生きているのに animation は nil
XCTAssertTrue failed - Outer scope's value-driven animation was stripped by the descendant scope boundary
```

トランザクションは Outer の stamp (animation 記述付き) を運んだまま、`transaction.animation` だけが nil になっている点が決定的。

## 機構

`AnimationScopeBoundaryModifier` は復元条件を stamp の **UUID 一致**に限定している:

```swift
// AnimationBarrier.swift
guard !transaction.disablesAnimations,
  incomingStamp?.id == stamp.id,
  let restoredAnimation = incomingStamp?.animation
else { return }
```

祖先スコープのトランザクションは (アニメーションの同異に関係なく) 子孫スコープの境界で常に破棄される。「AnimationScope blocks animations from ancestors」というドキュメント上の設計と整合的ではあるが、**「祖先 = 自分を包む別の AnimationScope」の場合もブロック対象になる**ことは README からは読み取れず、ネスト例・警告も存在しない。

## 提案 (優先度順)

1. **ドキュメント**: README にネスト時の意味論を明記する —「子孫スコープは祖先スコープの value 駆動アニメーションもブロックする。単一サブツリーに複数の (animation, value) ペアを掛けたい場合、ネストでは表現できない」。
2. **DEBUG 診断**: スタンプ済みアニメーションが「他スコープの境界」で strip されたときの runtime warning。現状の leak 検出は未スタンプのみが対象で、この消失モードは完全に不可視 (今回それが実害になった)。
3. **API 拡張の検討**: 単一スコープで複数トリガーを持てる形。例:
   ```swift
   AnimationScope(name: "Board") {
     ...
   }
   .scopeAnimation(.easeOut(duration: 0.12), value: selectedPoints)
   .scopeAnimation(.spring(...), value: hintPoints)
   ```
   1 つの stamp / boundary を共有する複数の value トリガーが持てれば、今回のケース (盤面の選択とヒント) はネストなしで表現できる。

## 実環境での回避策 (参考)

NumPath 側は該当サブツリーのみ生の `.animation(_:value:)` ×2 に戻し、行末マーカー + アーキテクチャ検査スクリプトの許可フィルタで「本 Issue 解決までの例外」として管理する。

## 付録 B: Issue #2 全文

> **Title**: ドキュメント提案: スコープ構成 (兄弟/ネスト) ガイド・静的 lint の具体レシピ・barrier の静止保証ユースケース
> **Labels**: documentation

## 背景

実アプリ (NumPath: SwiftUI パズル + AdMob バナー) に v0.1.0 を導入した。最終構成は:

- 兄弟サブツリーへの value 駆動 `AnimationScope` ×2 (クリア演出 scale / フィードバック行)
- 広告バナースロット (高さ予約 frame) の外側に `.animationBarrier()` — 「バナー領域はレイアウトシフト禁止」という既存規約の機械的裏付けとして機能し、テスト広告のクリエイティブローテーション 3 回・クリア演出・ヒント表示を通してヘッダー座標不変をシミュレータで確認できた
- root に DEBUG 限定 `detectAnimationLeaks()` — セッション全体で警告ゼロ = 移行漏れなしの証明として有効だった

導入自体は成功したが、README だけでは判断できず手探りになった点があったので、ドキュメント追加の提案。

## 提案

1. **構成ガイド (composition) の節を README か docs に追加する**
   - 兄弟スコープの例: 「隣接する 2 つのサブツリーにそれぞれ別の (animation, value) を掛ける」は最頻出の実戦形なのに例がない。
   - ネストの警告: 子孫スコープは祖先スコープのアニメーションもブロックする (#1 参照)。「同一サブツリー × 複数トリガー」はネストでは表現できないことを明記。
   - `animationBarrier()` と「高さ予約済みスロット」の組み合わせ例 (広告・レガシー UI の静止保証) は barrier の最も説得力ある用途なので Quick Start に足す価値がある。

2. **静的 lint の具体レシピを README に載せる**
   Detection Accuracy の節は「static review or a future lint rule for raw `.animation(` usage」と書いているが、具体例がない。今回は CI 用シェル検査に次を追加して運用した:

   ```bash
   # 生 animation API の禁止 (ScopedAnimation 経由に集約)。例外は行末マーカーで許可
   hits=$(grep -rn "withAnimation" Sources --include="*.swift" | grep -v "animation-exception:")
   hits=$(grep -rn "\.animation(" Sources --include="*.swift" | grep -v "animation-exception:")
   ```

   root detector の既知の盲点 (detector より下で生成される raw `.animation(_:value:)` トランザクション) はこの grep が実質的に埋めるので、「3 層」運用の 3 層目として具体形を示すと採用者が迷わない。

3. **導入判断の目安**: 「`withAnimation` が 0 で `.animation(_:value:)` が数箇所」のようなアプリでは blocking + detection の恩恵は主に (a) バナー等の静止保証と (b) 将来のリーク検出網になる。README の想定読者はリークに苦しむ大規模アプリに見えるが、小規模アプリでの採用価値 (構造の先行固定) も一節あると裾野が広がる。
